# WebSocket 与数据获取架构

## 1. WebSocket 用途

WebSocket **仅用于与前端实时交互**，不用于外部数据源接入。

| 通道 | 方向 | 推送内容 |
|------|------|----------|
| `/ws/market` | 服务端→前端 | 实时行情推送（tick/分钟线更新） |
| `/ws/portfolio` | 服务端→前端 | 持仓变动、盈亏更新 |
| `/ws/risk` | 服务端→前端 | 风控告警、规则触发通知 |
| `/ws/strategy` | 服务端→前端 | 策略状态变更（启动/停止/参数调整） |
| `/ws/backtest` | 服务端→前端 | 回测进度、结果通知 |
| `/ws/order` | 服务端→前端 | 订单状态变更（提交/成交/拒绝） |

所有 WebSocket 通道均为**服务端主动推送**，前端订阅后接收消息。前端不需要通过 WebSocket 发送控制命令（控制命令走 REST API）。

## 2. 外部数据获取

外部数据源（akshare 等）通过 **HTTP API 同步/异步调用** 获取，不经过 WebSocket。

```
┌─────────────┐     HTTP API      ┌──────────────┐
│  quant_invest │ ──────────────→  │  akshare 等   │
│  DataProvider  │ ←────────────── │  数据源       │
└─────────────┘     响应数据       └──────────────┘
```

### 数据获取流程

1. **定时任务触发**：`CronScheduler` 按配置触发数据拉取任务
2. **DataProvider 调用**：通过 HTTP 请求（akshare 的 Python API）获取行情/财务/宏观数据
3. **数据写入**：拉取的数据写入 `StorageEngine`（内存缓存 + 磁盘持久化）
4. **推送通知**：数据更新后，通过 EventBus 发布事件，WebSocket 服务端推送给前端

### 数据类型

| 数据类型 | 获取方式 | 更新频率 |
|----------|----------|----------|
| 日线行情 | HTTP API (akshare) | 每日收盘后 |
| 分钟行情 | HTTP API (akshare) | 盘中定时 |
| 财务数据 | HTTP API (akshare) | 季度报告期 |
| 宏观指标 | HTTP API (akshare) | 按发布周期 |
| 实时行情 | 待接入（券商接口） | Tick 级别 |

## 4. 后续扩展

- **券商实时行情**：未来接入券商 WebSocket/快速交易接口时，需新增独立的 `MarketDataReceiver` 组件，与前端 WebSocket 推送通道分离
- **io_uring 集成**：C++ 网络层协程化后，WebSocket 服务端可替换为基于 io_uring 的高性能实现
- **数据缓存策略**：高频数据（tick/分钟线）需在 `TimeSeriesCache` 中做 LRU 驱逐，低频数据（财务/宏观）可磁盘持久化
